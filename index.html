Pourquoi Utiliser Plusieurs Branches dans Git ?

  • Travail Simultané
  • Isolation des Modifications
  • Expérimentation Sûre
  • Gestion des Conflits
  • Stabilité du Code

Enregistrement et Envoi des Modifications

  1. Ajouter les Modifications git add <fichier>
  2. Créer un Commit git commit -m "Message de commit" , en respectant les conventionals commit
  3. Envoyer les Modifications git push origin <nom-branche> On push pas directement sur la develop!
Ne jamais push directement sur develop ! Toujours passer par une PR.

Structure d'un Commit Conventionnel

Un commit conventionnel suit cette structure :

<type>(<scope>): <description>

Exemple :

test(auth): ajout de tests pour la connexion utilisateur

Explication :

  • test : Indique qu'il s'agit de l'ajout de tests.
  • (auth) : Spécifie que les tests concernent l'authentification.
  • ajout de tests pour la connexion utilisateur : Brève description du changement.

les types de Commits

  • feat: Ajout d'une nouvelle fonctionnalité (feat: ajout du mode sombre)
  • fix: Correction de bug )
  • docs: Mise à jour de la documentation (docs: amélioration du README)
  • style: Changement de style
  • refactor: Refactorisation du code sans changement de fonctionnalité
  • test: Ajout ou mise à jour des tests

La Pull Request (PR)

  • Création via GitHub
  • En general, la PR est proposée automatiquement apres un push mais au besoin, Sélectionner la branche source et la branche cible (la develop)
  • Rédiger une courte description
  • Demander une revue de code en selectionnant 1 ou plsrs reviewers
Une PR permet de valider et de discuter les modifications avant de les intégrer. on peut aussi conditionner par la reussite de test, le reviewer peut approuvé, refuser, ou jouer la montre

Merge d'une Pull Request

  • Commit de merge : Fusion explicite avec l'historique complet
  • Rebase : Réécrit l'historique pour un historique linéaire
  • Squash : Fusionne plusieurs commits en un seul
git merge --no-ff feature/ma-feature
- **Commit de merge** : Ajoute un commit spécifique marquant la fusion, permettant de visualiser l'origine des changements. - **Rebase** : Réécrit l'historique pour le rendre linéaire, utile pour éviter les commits de merge inutiles mais risqué si conflits. - **Squash** : Condense tous les commits d'une branche en un seul, idéal pour une PR avec plusieurs petits commits. - Dans la plupart des cas, le **commit de merge** est privilégié car il garde l'historique clair et traçable. c celui utilisé par defaut quand on valide sur github

Mise à Jour des Branches

  • Après une Pull Request (PR), on detruit la branche
  • git pull origin develop si besoin de se mettre a jour et on tire une nouvelle branche depuis la develop
